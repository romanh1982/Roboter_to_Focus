# MiniVario Profile-Replay Test — Tool Description & Firmware Integration Guide

## 1. Purpose

This document describes:

1. The **`kuka_movement_viewer.py`** desktop GUI tool that extracts a focus-distance profile from KUKA robot programs
2. The **C++ lookup table (LUT)** it exports
3. How to integrate the LUT into the existing MiniVario low-speed test framework as a new **Profile-Replay** test mode (`LOWSPEED_TEST_PROFILE_REPLAY`)

The goal is to reproduce the exact distance-over-time behavior that the Leica AT960 laser tracker experiences during a real robot application, and to replay that profile through the MiniVario focus drives on the bench — without needing the robot or tracker present.

---

## 2. System Context

### 2.1 Real Application Setup

```
  KUKA Robot (moving TCP with scanner)
        │
        │  Euclidean distance d(t)
        ▼
  Leica AT960 Laser Tracker (fixed position)
        │
        │  inputZoomDist = d(t)  [mm]
        ▼
  MiniVario (adjusts optical focus via Zoom1 + Zoom2 carriages)
```

During live operation, the tracker continuously measures the distance to the scanner mounted on the robot TCP. This distance feeds into the MiniVario as `inputZoomDist`, which drives the two optical carriages via the zoom lookup tables (`zoomTableS8[]`, `zoomTableS23[]`).

### 2.2 Bench Test Replacement

The Profile-Replay test removes the robot and tracker from the loop. Instead, a pre-computed distance profile stored as a `const long[]` array is played back at the 100 Hz generator rate inside `SyncVarioTrkCyclicData()`. The MiniVario drives follow this profile exactly as they would during the real application.

```
  profileReplayLUT[]  (const long array, one entry per 10 ms cycle)
        │
        │  testDistanceRT = profileReplayLUT[index++]
        ▼
  SyncVarioTrkCyclicData()  →  inputZoomDist  →  CalcTrajectory()  →  MoveLensesRT()
```

## 3. The GUI Tool — `kuka_movement_viewer.py`

### 3.1 What It Does

The tool is a Python/Tkinter desktop application that:

1. **Parses** KUKA KRL `.src` and `.dat` file pairs to extract all waypoints with their Cartesian positions (X, Y, Z, A, B, C), movement types (SPTP/SLIN), velocities, and blending modes
2. **Computes** the Euclidean distance from a user-configurable laser tracker position to each waypoint
3. **Estimates** the time profile based on programmed velocities
4. **Resamples** the distance profile onto a uniform 100 Hz time grid matching the MiniVario distance generator rate
5. **Rate-limits** the resampled profile to ±60 mm/cycle (`MAX_DELTA_DIST_PER_CYCLE_POS_100`)
6. **Clamps** all values to the MiniVario input range (1500 mm .. 25000 mm)
7. **Exports** the result as a C++ header file containing a `const long[]` lookup table

### 3.2 Requirements

- Python 3.8+
- `matplotlib` (for plotting)
- `tkinter` (usually included with Python)

### 3.3 Usage

```bash
python3 kuka_movement_viewer.py
```

1. Click **"Load .src / .dat files"** and select the KUKA `.src` file (the tool auto-finds the matching `.dat`)
2. Set the **Tracker position** (X, Y, Z) in mm — this is the position of the AT960 in the robot's base coordinate frame
3. Click **"Recalculate"** to recompute the distance profile
4. Review the 5 tabs:
   - **Robot Path** — 2D projections (XY, XZ, YZ) with tracker position overlay
   - **Tracker Distance vs Time** — raw distance profile with MiniVario range band
   - **MiniVario Focus Profile** — clamped distance + change rate
   - **100 Hz LUT Preview** — the final resampled array as it will appear in C++
   - **Data Table** — all waypoints with computed distances
5. Click **"C++ Header (Profile Replay LUT)"** to export `VarioProfileReplayLUT.h`

### 3.4 Tracker Position

The tracker position is not stored in the KUKA program files. It must be obtained from the cell layout or measured. The tool provides three entry fields (X, Y, Z) in the robot's Base[0] coordinate system.

If the exact position is unknown, a reasonable estimate can be derived from the simulation image: the tracker is typically placed 1–3 meters from the robot base, at floor level or on a tripod.

### 3.5 Export Formats

| Export | Description | Use Case |
|--------|-------------|----------|
| **CSV (full data)** | All waypoints with positions, velocities, distances, timing | Analysis in Excel |
| **CSV (MV profile)** | Time + raw distance + clamped distance | External test generators |
| **C++ Header (LUT)** | `const long profileReplayLUT[]` with `#define` constants | Firmware integration |

## 4. The Exported C++ Header

### 4.1 File Structure

The exported file `VarioProfileReplayLUT.h` contains:

```c
// Auto-generated by kuka_movement_viewer.py  (2026-03-24 14:30:00)
// Source program: Tool_1_Base_0.src
// Tracker position: X=0.0  Y=-1800.0  Z=0.0 mm
// Generator rate: 100 Hz  (10 ms / cycle)
// Distance range: 1500 .. 3827 mm
// Duration: 5219 cycles = 52.19 s
// Rate-limited to max 60 mm/cycle

#ifndef VARIO_PROFILE_REPLAY_LUT_H_
#define VARIO_PROFILE_REPLAY_LUT_H_

#define PROFILE_REPLAY_LUT_SIZE   5219
#define PROFILE_REPLAY_CYCLE_MS   10

// Distance values in mm, one entry per 100 Hz generator cycle.
// Usage: testDistanceRT = profileReplayLUT[profileReplayIndex];
const long profileReplayLUT[PROFILE_REPLAY_LUT_SIZE] = {
   1861,  1852,  1842,  1833,  1823,  1813,  1804,  1794,  1785,  1775,
   1766,  1756,  1746,  1737,  1727,  1718,  1708,  1698,  1689,  1679,
   ...
};

#endif // VARIO_PROFILE_REPLAY_LUT_H_
```

### 4.2 Data Guarantees

The tool guarantees the following properties for every exported LUT:

| Property | Guarantee | Enforced by |
|----------|-----------|-------------|
| Value range | Every entry ∈ [1500, 25000] | `clamp_mv()` |
| Rate limit | \|LUT[i] − LUT[i−1]\| ≤ 60 mm | Post-processing loop |
| Timing | One entry per 100 Hz cycle (10 ms) | Resampling to uniform grid |
| Data type | `long` (32-bit signed integer) | Matches `testDistanceRT` type |
| Units | Millimeters | Matches `inputZoomDist` units |

## 5. Firmware Integration — Profile-Replay Test Mode

### 5.1 Architecture Overview

The Profile-Replay mode plugs into the existing low-speed test framework alongside the triangular-wave test and the step-and-settle test. It uses the same command interface, the same distance generator entry point, and the same motion execution pipeline.

```
LowSpeedTestCommandId                  SyncVarioTrkCyclicData()
─────────────────────                  ────────────────────────
cmd_lowSpeedTest_testOff          ──►  (clears all flags)
cmd_lowSpeedTest_startSensorRange ──►  LOWSPEED_TEST_TRIANGLE
cmd_lowSpeedTest_startFullRange   ──►  LOWSPEED_TEST_TRIANGLE
cmd_lowSpeedTest_startStepSettle  ──►  LOWSPEED_TEST_STEP_SETTLE
cmd_lowSpeedTest_startProfileReplay ─► LOWSPEED_TEST_PROFILE_REPLAY  ◄── NEW
cmd_lowSpeedTest_checkStatus      ──►  (polls flags)
```

### 5.2 Required Changes

#### 5.2.1 Include the LUT header

In `VarioFunctions.cpp`, add:

```cpp
#include "VarioProfileReplayLUT.h"
```

#### 5.2.2 Extend the `LowSpeedTestMode` enum

In `VarioZoomDefs.h`, add the new mode to the existing enum:

```cpp
enum LowSpeedTestMode {
    LOWSPEED_TEST_NONE = 0,
    LOWSPEED_TEST_TRIANGLE,          // existing triangular-wave test
    LOWSPEED_TEST_STEP_SETTLE,       // step-and-settle test
    LOWSPEED_TEST_PROFILE_REPLAY     // NEW: replay recorded distance profile
};
```

#### 5.2.3 Add the profile-replay index counter

In `ServoDataSet` (in `VarioZoomDefs.h`), add:

```cpp
unsigned long profileReplayIndex;    // current index into profileReplayLUT[]
```

#### 5.2.4 Add the command

In `ExchangeVarioCommands.h`, extend the `LowSpeedTestCommandId` enum:

```cpp
enum LowSpeedTestCommandId
{
    cmd_lowSpeedTest_testOff = 1,
    cmd_lowSpeedTest_startSensorRange = 22,
    cmd_lowSpeedTest_startStepSettle = 2,
    cmd_lowSpeedTest_startFullRange = 3,
    cmd_lowSpeedTest_startProfileReplay = 4,   // NEW
    cmd_lowSpeedTest_checkStatus = 10,
};
```

#### 5.2.5 Command handler — `VarioCmdLowSpeedTest.cpp`

Add a new case in the `switch (lowSpeedTestCmd)` block of `LowSpeedTestOnOff()`:

```cpp
case cmd_lowSpeedTest_startProfileReplay:
    if (pDriveDataPool->trajectoryDataSet.zoomInitOkFlag == false)
        status = error_lowSpeedTest_varioNotInit;
    else if (pDriveDataPool->servoDataSet.rtTrackingFlag == true)
        status = error_lowSpeedTest_varioIsBusy;
    else
    {
        // Select mode
        pDriveDataPool->servoDataSet.lowSpeedTestMode = LOWSPEED_TEST_PROFILE_REPLAY;

        // Reset replay index
        pDriveDataPool->servoDataSet.profileReplayIndex = 0;

        // Standard test setup
        pDriveDataPool->servoDataSet.lowSpeedTestDurationCtr = 0;
        pDriveDataPool->servoDataSet.testSuccessfulFlag = false;
        pDriveDataPool->servoDataSet.testErrorOnDriveX = 0;
        pDriveDataPool->servoDataSet.startDistGeneratorFlag = true;
        pDriveDataPool->servoDataSet.trkSimInProgressFlag = false;

        status = state_lowSpeedTest_acknowledged;
    }
    break;
```

#### 5.2.6 Distance generator — `VarioFunctions.cpp`

Inside the `if (pDp->servoDataSet.startDistGeneratorFlag == true)` block in `SyncVarioTrkCyclicData()`, add a new branch in the `lowSpeedTestMode` switch:

```cpp
case LOWSPEED_TEST_PROFILE_REPLAY:
{
    // On first entry, move to start distance
    if (pDp->servoDataSet.trkSimInProgressFlag == false)
    {
        pDp->trajectoryDataSet.testDistanceRT = profileReplayLUT[0];
        pDp->servoDataSet.profileReplayIndex = 0;
        pDp->servoDataSet.trkSimInProgressFlag = true;
    }

    // Advance index at 100 Hz rate (each call = one cycle)
    if (pDp->servoDataSet.profileReplayIndex < PROFILE_REPLAY_LUT_SIZE)
    {
        pDp->trajectoryDataSet.testDistanceRT =
            profileReplayLUT[pDp->servoDataSet.profileReplayIndex];

        pDp->servoDataSet.profileReplayIndex++;
        pDp->servoDataSet.lowSpeedTestDurationCtr++;
    }
    else
    {
        // Profile complete
        pDp->servoDataSet.testSuccessfulFlag = true;
        pDp->servoDataSet.startDistGeneratorFlag = false;
    }

    // Feed to trajectory (same as existing tests)
    pDp->trajectoryDataSet.inputZoomDist =
        (long)pDp->trajectoryDataSet.testDistanceRT;
    break;
}
```

### 5.3 No Other Changes Required

The profile replay mode requires **no modifications** to:

- `CalcTrajectory()` — it receives `inputZoomDist` as usual
- `MoveLensesRT()` — servo loop runs unchanged
- `VarioThread.cpp` — error detection via `CheckPositionErrorDriveX()` works as-is
- `cmd_lowSpeedTest_testOff` — existing off-handler clears `startDistGeneratorFlag`
- `cmd_lowSpeedTest_checkStatus` — existing status handler checks `testSuccessfulFlag` and `testErrorOnDriveX`

### 5.4 Memory Considerations

| Parameter | Typical Value |
|-----------|---------------|
| LUT size (this example) | 5219 entries |
| Memory footprint | 5219 × 4 bytes = 20,876 bytes ≈ 20 KB |
| Duration | 52.2 seconds |
| Distance range | 1500 – 3827 mm |

The LUT is declared `const long[]`, so it resides in flash/ROM, not in RAM. For longer robot programs or higher resolution, larger LUTs can be generated. A 60-second profile at 100 Hz requires ~24 KB.

### 5.5 Execution Flow

```
AT-Inspect sends:  cmd_vario_low_speed_test  with  cmd_lowSpeedTest_startProfileReplay
        │
        ▼
LowSpeedTestOnOff()
  ├── check zoomInitOkFlag
  ├── set lowSpeedTestMode = LOWSPEED_TEST_PROFILE_REPLAY
  ├── reset profileReplayIndex = 0
  └── set startDistGeneratorFlag = true
        │
        ▼
SyncVarioTrkCyclicData()  @ 100 Hz
  ├── testDistanceRT = profileReplayLUT[profileReplayIndex++]
  ├── inputZoomDist = testDistanceRT
  └── if (profileReplayIndex >= PROFILE_REPLAY_LUT_SIZE):
        testSuccessfulFlag = true
        │
        ▼
CalcTrajectory()
  ├── lookup zoomTableS8[index]   → nextTargetLensPos[ZOOM1]
  └── lookup zoomTableS23[index]  → nextTargetLensPos[ZOOM2]
        │
        ▼
MoveLensesRT()  @ 200 Hz
  └── PID control drives both carriages to target positions
```

### 5.6 Observability

All existing telemetry signals work without modification:

| Signal | Content |
|--------|---------|
| `inputZoomDist` | Current distance target from LUT (mm) |
| `currentLensPos[0/1]` | Actual lens positions (increments) |
| `controlErrorNew[0/1]` | Position error per lens (increments) |
| `lowSpeedTestDurationCtr` | Cycles elapsed |
| `testSuccessfulFlag` | Set `true` when LUT is fully played |
| `testErrorOnDriveX` | Set by `VarioThread` if a drive error occurs |

Additionally, `profileReplayIndex` can be exposed via the debug scope server if needed, to show playback progress.

## 6. Testing the Integration

### 6.1 Basic Validation

1. Flash firmware with the new code and LUT header
2. Initialize the MiniVario (`cmd_vario_initialize`)
3. Send `cmd_lowSpeedTest_startProfileReplay`
4. Monitor `inputZoomDist` in AT-Inspect — it should follow the LUT profile
5. Monitor `controlErrorNew[0/1]` — both should stay within tolerance
6. Poll `cmd_lowSpeedTest_checkStatus` — should return `state_lowSpeedTest_testSuccessful` after ~52 s

### 6.2 What to Look For

- **Drive stalls**: The profile includes both fast transitions (SPTP at 100%) and slow precision moves (SLIN at 0.08 m/s). If a commutator dead-zone causes a stall during a slow section, it will show as a growing `controlErrorNew` and eventually trigger `testErrorOnDriveX`.
- **Direction reversals**: The robot path causes the tracker distance to oscillate, creating frequent direction changes in the MiniVario drives. These are the critical moments for detecting stick-sleep issues.
- **Speed variation**: Unlike the uniform-speed triangular test, this profile has realistic speed variation — exactly the conditions where commutator problems manifest.

### 6.3 Comparison with Existing Tests

| Aspect | Triangular Test | Step-and-Settle | Profile Replay |
|--------|----------------|-----------------|----------------|
| Motion pattern | Constant-speed ramp | Discrete steps | Recorded real profile |
| Speed profile | Uniform (~20 mm/s) | Step + settle | Variable (real-world) |
| Direction changes | 1 (at top) | At each step | Many (as in real app) |
| Distance range | 1.5 m – 25 m | 1.5 m – 25 m | Application-specific |
| Duration | ~20 min (full range) | ~30 min | Application-specific (~52 s) |
| Detects | Basic drive function | Settling behavior | Real-world failure modes |

## 7. Generating LUTs for Different Applications

To create a profile for a different robot program or tracker position:

1. Obtain the `.src` and `.dat` files from the KUKA controller
2. Determine the tracker position in the robot's base frame
3. Run the GUI tool, load the files, set the tracker position
4. Review the LUT preview tab — check that the distance range and rate limits are reasonable
5. Export the C++ header
6. Replace `VarioProfileReplayLUT.h` in the firmware project and rebuild

Multiple profiles can be maintained as separate header files (e.g., `VarioProfileReplayLUT_App1.h`, `VarioProfileReplayLUT_App2.h`) and selected via `#include` at build time, or — for runtime selection — stored as separate arrays and switched via an additional command parameter.

## 8. Limitations and Assumptions

1. **Time estimation is approximate.** The tool estimates segment times from programmed velocities and straight-line distances. Real robot timing depends on acceleration profiles, path blending, and controller cycle times. For higher fidelity, actual robot timestamps from a trace recording should be used.

2. **PTP velocity mapping.** PTP moves are specified as % of maximum joint speed, which does not directly correspond to TCP Cartesian speed. The tool uses a rough mapping of 100% ≈ 2000 mm/s TCP speed. This is adequate for profile shape but not for exact timing.

3. **Tracker position is static.** The tool assumes the tracker does not move during the robot program. If the tracker is on a moving platform, the distance computation would need extension.

4. **Single LUT at compile time.** The current integration compiles one LUT into the firmware. For runtime-switchable profiles, the architecture would need extension (e.g., profile selection via command parameter, or profiles stored in EEPROM).
